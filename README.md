# Adaprive Learning Navigator

## О проекте

Adaprive Learning Navigator — это веб-сервис для изучения IT-направлений в формате roadmap-first.

Текущий вектор проекта:
- пользователь выбирает готовое IT-направление из каталога roadmap'ов;
- сервис показывает структуру тем, зависимости и материалы;
- на основе лимита часов в неделю и уже известных тем строится личный недельный план;
- AI используется не как основной генератор roadmap, а как будущий помощник по конкретной теме.

По сути, проект движется в сторону аналога `roadmap.sh`, но:
- с русскоязычным интерфейсом;
- с личным недельным темпом обучения;
- с учётом уже знакомых тем;
- с перспективой AI Tutor внутри темы.

## Что уже реализовано

### Пользовательский функционал

- регистрация, вход и обновление JWT-токенов;
- защищённая приватная часть приложения;
- многостраничный frontend внутри Spring Boot;
- единый каталог roadmap'ов с переходом к отдельной карте выбранного направления;
- просмотр деталей темы, её зависимостей и ресурсов;
- построение личного плана по roadmap на основе `hoursPerWeek`;
- если одна тема по оценке длиннее недельного лимита пользователя, сервис автоматически дробит её на несколько последовательных недельных шагов вместо ошибки;
- на фронте такие разрезанные шаги помечаются как `Часть 1 из N`, `Часть 2 из N`, а также получают мягкие подписи ритма: `Старт темы` для первой части и `Продолжение темы` для следующих в weekly plan;
- прогресс прохождения встроен в недельный план и сохраняется на backend;
- просмотр сохранённых планов в профиле;
- страница недельного плана с интерактивным прогрессом;
- профиль с аккаунтом, датой регистрации и историей планов;
- legacy-режим AI-генерации плана всё ещё сохранён в backend, но это уже не основной сценарий продукта.

### Реализованные разделы интерфейса

#### Главная `/`
- кратко объясняет, что это за сервис;
- показывает основную пользу для нового пользователя;
- в hero-блоке оставлена одна основная кнопка без дублирующих чёрных CTA;
- нижний блок "Что получает пользователь" оставлен чисто информационным, без дублирующих CTA-кнопок;
- ведёт к регистрации или входу.

#### Вход `/login`
- авторизация по email и паролю;
- лаконичная русскоязычная entry-страница без технических формулировок и шаблонного текста;
- после успеха переводит пользователя в кабинет.

#### Регистрация `/register`
- создание аккаунта;
- лаконичная русскоязычная entry-страница в том же тоне, что и экран входа;
- после успешной регистрации пользователь получает токены и доступ к приватной части.

#### Кабинет `/dashboard`
- стартовая точка после входа;
- является главным разделом выбора направления и сборки персонального плана;
- показывает каталог roadmap'ов по двум вкладкам: role-based и skill-based;
- в каталоге выводится только MVP-набор направлений из базовой витрины `roadmap.sh`, без дополнительных beginner/legacy/экспериментальных карт;
- позволяет выбрать направление и открыть его отдельную карту через кнопку `К карте`;
- позволяет указать часов в неделю;
- позволяет отметить уже известные темы;
- в приватной части боковая панель навигации может сворачиваться и разворачиваться, а её состояние запоминается локально в браузере;
- каталог roadmap'ов сделан компактнее: на широком экране карточки направлений показываются плотнее и в несколько колонок;
- панель сборки персонального плана закреплена рядом с каталогом, чтобы пользователь не прокручивал страницу обратно вверх после выбора направления;
- если для выбранного направления уже есть план, кнопка `Открыть план` появляется прямо в блоке `Персональный план`;
- строит и сохраняет личный план по выбранному roadmap.

#### Roadmap `/roadmap`
- отдельная страница выбранного направления;
- открывается из каталога по кнопке `К карте` с query-параметром `roadmapId`;
- показывает сводку roadmap и roadmap-like граф тем как полноценное рабочее полотно на всю доступную ширину;
- связи между темами на карте сделаны более контрастными: центральная ось, ответвления и маркеры шагов визуально читаются как маршрут, а не как отдельные карточки;
- детали темы больше не занимают постоянную правую колонку: при выборе темы справа выезжает drawer-панель;
- в drawer-панели доступны обзор темы, ресурсы и placeholder под будущий AI Tutor;
- если для направления уже есть сохранённый weekly plan, со страницы roadmap можно сразу перейти в этот план;
- если плана ещё нет, вместо перехода в план показывается действие `Собрать план`, которое возвращает пользователя в каталог с выбранным направлением.

#### План `/plan`
- показывает сохранённый личный план по неделям;
- отображает часы, темы недели и объяснение порядка тем;
- теперь является основным местом работы с прогрессом: у каждой темы есть статусы `Не начато`, `В процессе`, `Готово`;
- сверху показывает сводку прогресса: процент завершения, закрытые темы, закрытые часы и следующий шаг;
- прогресс сохраняется на backend в `topic_progress` и привязан к конкретному плану пользователя;
- темы, отмеченные как уже известные, не попадают в weekly plan, но остаются видимыми в самом roadmap.
- верхняя карточка `Недельный ритм` сведена к краткой сводке без длинного поясняющего текста.
- в верхней карточке плана явно показывается исходный недельный лимит пользователя.

#### Профиль `/profile`
- показывает только пользовательски полезную информацию: аккаунт, дату регистрации, количество сохранённых планов и список планов;
- не выводит служебные данные сессии вроде internal user id, JWT payload или сроков действия токенов;
- не содержит старые legacy-поля `Цель`, `Уровень`, `Известные темы` и дефолтные часы, потому что эти параметры больше не участвуют в roadmap-first сценарии;
- показывает список сохранённых weekly plan с переходами в каталог, карту направления и сам план.

#### Прогресс `/progress`
- больше не является отдельным основным разделом;
- маршрут сохранён как compatibility-route и перенаправляет пользователя в `/plan`, потому что прогресс встроен прямо в недельный план.

## Текущая продуктовая модель

Сейчас в проекте существуют два сценария:

### 1. Основной сценарий
Roadmap-first:
- выбрать roadmap;
- просмотреть структуру тем;
- исключить уже известные темы;
- построить план по неделям на основе количества часов.

### 2. Legacy-сценарий
AI-generated route:
- запросить генерацию плана через AI;
- backend валидирует результат и сохраняет план.

Этот сценарий пока оставлен в проекте, но больше не является основной логикой интерфейса.

### Конфигурация

Основные настройки находятся в:
- `src/main/resources/application.yaml`

Что важно:

- H2 используется как файловая база;

- обычный запуск приложения не делает массовый импорт roadmap'ов на старте;
- браузер может открываться автоматически при старте;
- для AI используется Ollama и модель `qwen2.5:7b`.

### Запуск

Варианты:
- запуск из IDE;
- запуск через Maven.

После старта приложение открывает сайт на главной странице автоматически, если это разрешено окружением и настройками.
Автооткрытие браузера работает в best-effort режиме: как только web-сервер поднимает порт, приложение пробует открыть страницу через `Desktop.browse`, а на Windows дополнительно использует fallback через `cmd /c start`, `powershell Start-Process` и `explorer.exe`.

### Разовая загрузка KB из roadmap.sh

Полное наполнение локальной H2-базы делается отдельно от обычного старта приложения.

Для этого есть bootstrap-профиль:
- `src/main/resources/application-roadmap-bootstrap.yaml`

И готовый Windows-скрипт:
- `tools/Bootstrap-RoadmapKnowledgeBase.ps1`

Что делает bootstrap:
- запускает приложение в `web-application-type=none`;
- не поднимает web-сервер и не открывает браузер;
- выполняет одноразовый массовый import roadmap-данных в локальную H2-базу;
- после завершения завершает процесс.

Обычный рабочий режим после bootstrap:
- база уже наполнена;
- каталог читается сразу из H2;
- runtime sync на старте приложения отключён.

## roadmap.sh catalog import

В проекте появился первый технический слой для системного расширения базы знаний по публичному каталогу `roadmap.sh`.

Правило локализации для import-слоя:
- весь пользовательский описательный контент в manifest-файлах должен храниться на русском языке;
- транслит в описаниях запрещён;
- английский сохраняется только там, где это официальное название технологии, фреймворка, roadmap-направления или внешнего ресурса.
- при обновлении такого контента фронтенд должен сбрасывать устаревший localStorage-кеш roadmap-данных, чтобы пользователь сразу видел актуальные русские описания.
- это правило распространяется и на уже существующие legacy KB-данные проекта, включая `java-backend` и `java-mobile`.

Что уже реализовано:
- classpath-manifest с каталогом направлений: `src/main/resources/roadmap-sh/catalog.json`;
- загрузчик каталога: `ClasspathRoadmapShCatalogSource`;
- сервис синхронизации в `role_goals`: `RoadmapShCatalogSyncService`;
- стартовый runner для dev-синхронизации при запуске приложения: `RoadmapShCatalogSyncRunner`;
- отдельные тесты на загрузку manifest-файла и на upsert-сценарий каталога.

Как это работает сейчас:
- импорт по умолчанию выключен;
- при включении синхронизации новые направления создаются в `role_goals` со статусом `DRAFT`, чтобы не ломать текущий пользовательский каталог;
- существующие записи могут обновлять `name` и `description`;
- таблицы `topics`, `role_topics`, `topic_prereqs`, `resources` и `topic_resources` пока этим слоем не наполняются.

Конфигурация:
- `app.roadmap-sh.catalog-sync.enabled`
- `app.roadmap-sh.catalog-sync.catalog-location`
- `app.roadmap-sh.catalog-sync.default-status`
- `app.roadmap-sh.catalog-sync.update-existing`

Зачем это нужно:
- перестать расширять knowledge base вручную разрозненными SQL-вставками;
- зафиксировать единый вход для массового наполнения roadmap-каталога;
- подготовить безопасную основу для следующего шага: импорта тем, связей, ресурсов и подтем.

Следующий технический шаг для этого слоя:
- добавить import topics-level данных поверх catalog-level синхронизации;
- связать импортируемые темы с `role_goals` через `role_topics`;
- отдельно продумать стратегию хранения русских локализаций названий и описаний.

## roadmap.sh content import

Поверх каталога теперь работает полноценный content-level import roadmap-данных.

Что реализовано сейчас:
- в `src/main/resources/roadmap-sh/roadmaps/*.json` хранится полный набор manifest-файлов для roadmap-направлений из `roadmap.sh`;
- import покрывает современный формат `nodes/edges`, legacy-формат `mockup.controls.control` и content-only roadmap-директории;
- в content-sync попадают:
  - `role_goals`
  - `topics`
  - `role_topics`
  - `topic_prereqs`
  - `resources`
  - `topic_resources`
- для локального обновления import-слоя добавлен генератор:
  - `tools/Generate-RoadmapShManifests.ps1`
  - `tools/roadmap-sh-localization.json`

Итог текущего bulk-import слоя:
- `87` manifest-файлов roadmap-направлений;
- `9210` тем и подтем;
- `20612` ресурсных ссылок;
- полное покрытие всех каталогов из `roadmap.sh`, доступных в локальном upstream-репозитории.

Как устроена локализация:
- user-facing описания направлений и тем генерируются на русском;
- транслит в описаниях запрещён;
- названия технологий, внешних ресурсов и самих roadmap-направлений могут оставаться в оригинале;
- curated manifest-файлы (`game-developer`, `java-backend`, `java-mobile`) bulk-генератор не перезаписывает.
- после обновления bulk-import слоя фронтенд сбрасывает версионированный roadmap-кеш (`session.js`), чтобы каталог и описания перечитывались из backend, а не из устаревшего `localStorage`.

Как устроена генерация:
- источник данных: локальный clone `roadmap.sh` в `.tmp/roadmap-sh-upstream`;
- для новых roadmap-графов темы и связи берутся из `nodes/edges`;
- для legacy-roadmap используется `migration-mapping.json` и иерархия `controlName`;
- для content-only roadmap темы строятся напрямую по markdown-файлам из `content`;
- ресурсы извлекаются из markdown-списков и связываются с темами автоматически;
- `catalog.json` пересобирается из manifest-слоя, чтобы catalog-sync и content-sync были согласованы между собой.

Конфигурация content-sync:
- `app.roadmap-sh.roadmap-sync.enabled`
- `app.roadmap-sh.roadmap-sync.manifest-pattern`
- `app.roadmap-sh.roadmap-sync.update-existing`
- `app.roadmap-sh.bootstrap.enabled`
- `app.roadmap-sh.bootstrap.catalog-sync`
- `app.roadmap-sh.bootstrap.roadmap-sync`

Dev-особенность import-слоя:
- в локальном запуске roadmap-sync читает manifest-файлы сначала напрямую из `src/main/resources/roadmap-sh/roadmaps`, а затем из classpath;
- это защищает dev-режим от ситуации, когда IDE не успела перекопировать новые JSON-ресурсы в compiled output, и каталог продолжает видеть старый набор roadmap'ов.
- если один и тот же roadmap встречается и в `file:`, и в `classpath:`, загрузчик схлопывает дубликаты по `roleCode` и оставляет первую найденную версию; это даёт приоритет локальным dev-manifest'ам и не ломает старт приложения.
- если направление раньше появилось через `catalog-sync` в статусе `DRAFT`, последующий `roadmap-sync` переводит его в `ACTIVE`, чтобы оно становилось видимым в пользовательском каталоге.
- перед content-sync сервис выравнивает identity-счётчики H2 для `role_goals`, `topics`, `topic_prereqs` и `resources`, чтобы bulk-import не падал на dev-базе после ручных seed-вставок с фиксированными `id`.
- content-sync больше не держит весь импорт одной общей транзакцией: изменения становятся видимыми по мере обработки roadmap'ов, а каталог не ждёт завершения всего bulk-import.
- user-facing API не показывает технические подписи о происхождении контента из `roadmap.sh`: такие пометки вычищаются из описаний и скрываются из resource metadata на уровне roadmap service.
- поверх импортированной KB работает отдельный слой пользовательской локализации: названия направлений, повторяющиеся англоязычные названия тем и формульные описания адаптируются на русский язык в API-ответах, при этом канонические названия технологий (`Java`, `Python`, `Git`, `Maven` и т.д.) сохраняются.
- слой локализации расширяется по частотному анализу импортированных roadmap: отдельно покрываются типовые темы вроде базового синтаксиса, алгоритмов поиска/сортировки, тестирования, деплоя, архитектурных паттернов, контейнеризации, очередей, ML-разделов и других повторяющихся заголовков.
- каталог направлений разделён по модели `roadmap.sh` на две группы: `Направления по ролям` (`ROLE_BASED`) и `Направления по навыкам` (`SKILL_BASED`). Категория вычисляется на read-model уровне по коду roadmap, поэтому БД и preloaded KB не требуют миграции.
- на странице каталога эти группы показываются как две вкладки: пользователь видит только выбранную категорию, а не две длинные секции подряд.
- поверх полного импортированного набора roadmap работает MVP-фильтр видимости: в публичном каталоге остаются только направления из актуальной витрины `roadmap.sh`, показанные в базовых role-based и skill-based списках. Это скрывает лишние beginner/legacy/экспериментальные карты без удаления данных из БД.
- тот же MVP-фильтр продублирован на frontend-слое каталога и выбора активной roadmap, чтобы старый API-ответ, stale build или сохранённый `localStorage` не возвращали скрытые направления в интерфейс.
- витринные названия role-based направлений дополнительно адаптированы как русские профессиональные роли: `DevOps-инженер`, `ML-инженер`, `Администратор PostgreSQL`, `AI-специалист по Data Science`; legacy-направления `java-backend` и `java-mobile` остаются классифицированы как role-based, но скрыты из MVP-каталога.
- для role-based направлений расширен слой локализации topic title: добавлены точные переводы, фразовый словарь и безопасные динамические правила для Frontend, Backend, DevOps, DevSecOps, Data/BI, AI/ML, Android/iOS, QA, Architecture, Cyber Security, UX, Technical Writing, GameDev, Product/Engineering Management и DevRel; технические названия технологий при этом сохраняются.
- отдельно закрыт список тем `AI-инженер` и усилены оставшиеся role-based темы: ML-математика, Android/iOS platform topics, DevSecOps/Cyber Security, PostgreSQL DBA, Product/Engineering Management и DevRel получают русские user-facing названия; вопросительные заголовки вроде `What is Data Warehouse?` нормализуются в русские вопросы.
- для skill-based направлений расширен тот же слой локализации: добавлены переводы для language roadmap'ов и базовых инженерных карт (`JavaScript`, `TypeScript`, `Java`, `Python`, `Go`, `Kotlin`, `Rust`, `C++`, `PHP`, `Ruby`, `SQL`, `HTML`, `CSS`, `Node.js`, `Git/GitHub`, `Linux`, `Shell/Bash`) и повторяющихся skill-topic'ов из Angular, System Design, AWS, Terraform и смежных roadmap'ов.
- отдельным вторым проходом усилена локализация наиболее шумных skill-based roadmap'ов: `JavaScript`, `Git/GitHub`, `Node.js`, `Linux`, `SQL`, `C++`, а также смежных language roadmap'ов. Пользовательские темы вроде `History of JavaScript`, `Cloning Repositories`, `History of Node.js`, `Why use C++?`, `Case Statements`, `Semantic Markup`, `Query Optimization` и аналогичные теперь отдаются API на русском.
- отдельным остаточным аудитом по `skill-based` roadmap'ам дочищены частотные общие user-facing темы, которые ещё оставались на английском: структуры данных и алгоритмы (`Stack`, `Queue`, `Graph`, `Tree`, `Full Binary Tree`, `Logarithmic`, `Big O`, `Adjacency List`, `Insertion Sort`), общие UI/framework-заголовки (`Selector`, `Template`, `Styles`, `App Lifecycle`, `App Architecture`, `Module Architecture`), повторяющиеся backend/design-system темы (`Object Relational Mapping`, `Triggers`, `Basic Routes`, `Basic Controllers`, `Governance`, `Need of Design System`) и часть общих тем из `WordPress` / `AI Agents` (`Activity Logging`, `Admin Menu`, `Embeddings and Vector Search`, `Observation & Reflection`). Оставшийся английский теперь в основном сводится к каноническим названиям технологий, библиотек, CLI-команд, framework-API и problem-name заголовкам.
- отдельным ручным проходом усилена локализация трёх сложных skill-based разделов: `Scala`, `SwiftUI` и `ASP.NET Core`. Для них переведены полуспециализированные user-facing темы вроде `The apply method`, `Case classes`, `Effect systems`, `Automatic Reference Counting (ARC)`, `Asynchronous Functions`, `Property Observers`, `ASP.NET Core Basics`, `Filters and Attributes`, `Distributed Cache`, `Task Scheduling` и другие аналогичные заголовки, при этом канонические имена библиотек и API (`Akka HTTP`, `CloudKit`, `Entity Framework Core`, `Dapper`) сохранены без насильственного перевода.
- для skill-based KB сохраняются канонические технические идентификаторы там, где перевод ухудшает узнаваемость upstream-объекта: названия языков, фреймворков, CLI-команд, package/tool имена и часть problem-name/command-name заголовков остаются в оригинале.
- версия frontend roadmap-кеша поднята до `2026-04-29-mvp-roadmap-catalog-v1`, чтобы браузер не показывал старые topic title и скрытые roadmap из `localStorage`.
- технические идентификаторы тем и направлений (`topicCode`, `roleCode`, `RM_...`) не должны выводиться в пользовательском интерфейсе: они остаются только во внутренних API-данных, поиске и fallback-сценариях, если у записи по какой-то причине нет человекочитаемого названия.
- на странице каталога/личного плана список `Уже знакомые темы` сделан компактнее: он рендерится в две колонки на desktop, имеет ограниченную высоту и внутренний scroll, а сами элементы уменьшены по вертикальному паддингу и типографике. Это позволяет не уводить кнопку построения плана далеко вниз на длинных roadmap.
- в том же списке сохранена читаемая последовательность roadmap: темы перед рендером сортируются по `priority`, а каждая карточка получает явный маркер шага (`Шаг N`). Это снимает проблему, когда в компактной двухколоночной раскладке визуально теряется порядок прохождения тем.

Текущий рекомендуемый режим работы:
- `roadmap-sync` и `catalog-sync` на обычном старте приложения выключены;
- для полного наполнения KB используется отдельный bootstrap-профиль;
- после завершения bootstrap база считается предзагруженной и используется как основной источник каталога и roadmap-данных.

Текущее ограничение слоя:
- import остаётся additive и safe-by-default;
- он не удаляет старые темы, связи или ресурсы, если они уже были заведены раньше вручную;
- русские описания пока генерируются по шаблонам, а не через отдельную модель переводов по каждому upstream-абзацу.

Следующие логичные шаги после наполнения KB:
- проверить, как массово импортированные roadmap'ы влияют на построение личного плана;
- при необходимости сузить weekly plan до core/required тем и отдельно показывать расширенные ветки roadmap;
- улучшить локализацию витрины ресурсов и человекочитаемых названий для отдельных content-only roadmap'ов.
